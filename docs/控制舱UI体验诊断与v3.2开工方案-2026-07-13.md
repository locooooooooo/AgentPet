# 控制舱 UI 体验诊断与 v3.2 开工方案

> 日期：2026-07-13
> 对象：`桌面牧场 · 控制舱`
> 结论性质：体验审计与实施输入，不代表产品代码已修改
> 当前基线：v3.1 已于 `aa4cfa5` 接受并推送

## 1. 一句话目标

让操作者在一个首屏内完成“识别异常 -> 选择 Agent -> 下发任务 -> 查看结果”，同时把治理明细、历史事件和重复状态降到按需查看层，避免控制舱变成状态文案陈列页。

## 2. 核心用户与最高频任务

核心用户是每天多次进入控制舱、管理 8 个 Agent 的单人操作者，而不是首次访问的浏览者。

最高频任务按优先级排序：

1. 发现当前是否有 Agent 在运行、失败或需要人工处理。
2. 快速选择一个 Agent，并确认它的身份、状态和当前任务。
3. 下发一个快捷任务或自定义任务。
4. 查看任务队列、最新结果和日志。
5. 在异常时查看 blocker、角色归属、Lane 和连接器限制。

## 3. 当前界面区域地图

| 区域 | 当前职责 | 主要组件/证据 |
| --- | --- | --- |
| Header + Status | 品牌、返回首页、3 个 KPI、连接器状态、最近事件、牧场设置、重置 | `NiuMaWorkspace.tsx` header；`StatusStrip.tsx` |
| 左监督轨 | 调度总览、blocker、角色、Lanes | `rail-summary-panel`、`orchestration-panel` |
| 中央工位 | 8 个 Agent 状态、选中、快捷动作、任务摘要 | `board-panel`、`agent-board`、`AgentCard` |
| 右 Operator | Agent 详情、quote、指标、任务/队列/日志 | `AgentDetailPanel` |
| Corner Assist | 当前 Agent 的快速状态和任务摘要 | `corner-assist` |
| Bottom Dock | 在线、运行、阻塞、最近更新时间 | `cockpit-dock` |

## 4. 证据等级

- **confirmed**：由当前源码、DOM 或 1280x720 / 1440x900 复放直接证明。
- **inferred**：由结构与高频工作流推导，实施前可用小范围验证确认。
- **needs replay**：需要真实 Electron、键盘或长期使用数据才能确认。

## 5. 关键痛点

### P0-1：核心派活路径在 1280x720 被上层详情挤压

**confirmed**

- 1280x720 下，右侧 tab 内容区仅约 `110px`。
- Agent 身份、quote、4 项指标和 tabs 均位于任务表单之前。
- “立即派活”存在但需要在右侧独立滚动区继续滚动才能到达；1440x900 下内容区约 `266px`，问题明显减轻。

影响：用户完成最高频任务前，先经过大量只读信息，形成“看详情优先、派活其次”的错误层级。

### P0-2：相同状态跨区域重复，增加扫读和口径成本

**confirmed**

- 在线、运行、阻塞同时出现在 Header KPI 与 Bottom Dock。
- latest event 同时出现在 StatusStrip、左轨“最新消息”和右侧 toast/事件信息。
- 选中 Agent 的名称、状态、头像、代号和任务摘要同时出现在中央卡、右详情与 Corner Assist。

影响：信息数量看似丰富，但决策信号没有增加；状态更新时间不一致时还会产生“哪个是真源”的疑问。

### P0-3：键盘焦点路径过长

**confirmed**

- 1280 当前 DOM 约有 `85` 个可聚焦项。
- 中央每个 Agent 卡同时暴露卡片、头像和 5 个动作。
- 右侧任务 Tab 约排到第 `75` 个焦点之后。
- 左右轨、折叠组和图标按钮进一步延长焦点路径。

影响：键盘用户难以快速到达“下发任务”，图标动作也缺少稳定的分组层级。

### P1-1：左监督轨把治理信息与现场操作放在同一视觉层级

**confirmed**

- 左轨同时承载 4 个摘要、长 blocker、35 个角色与 30 个 Lanes。
- 角色和 Lanes 默认折叠是正确方向，但标题、来源、目标和 blocker 仍持续占用首屏。
- blocker 目前已截断为三行，但仍与现场 Agent 异常争夺注意力。

影响：治理真源有价值，但不是每次派活都需要阅读全文。

### P1-2：中央 AgentCard 暴露过多同级动作

**confirmed**

- 每张卡包含选中入口、头像交互、状态/任务信息和 5 个动作按钮。
- 非选中卡仍保留完整动作集合，导致 8 卡矩阵产生大量小按钮和视觉噪声。

影响：用户很难在一眼内判断“点卡片选中”与“直接执行动作”的优先关系。

### P1-3：Corner Assist 与右详情重复

**confirmed**

- 两者都显示选中 Agent、状态、任务和指标。
- 右详情在桌面默认展开，Corner Assist 默认收起但仍有固定触发器。

影响：宽屏下重复；窄屏下 Corner Assist 可能有价值，应按布局条件保留而不是永久并存。

### P2-1：Header 文案与版本信息的决策价值低

**inferred**

- 品牌、版本和产品描述长期稳定，不参与日常调度决策。
- Header 已压缩，但仍使用较多水平空间承载静态说明。

建议：保留品牌和版本，将说明缩为 tooltip 或只在首次/帮助入口出现。

### P2-2：底部更新时间与重复 KPI 价值有限

**inferred**

- Dock 的在线/运行/阻塞与 Header 重复。
- “最近更新”有诊断价值，但更适合与数据新鲜度/连接状态组合，而不是单独占用大块。

## 6. 信息审计

| 信息/控件 | 处置 | 目标位置 | 理由与收益 | 风险 |
| --- | --- | --- | --- | --- |
| 品牌名称 | Keep | Header 左 | 提供产品身份 | 无 |
| 版本号 | Collapse | 品牌旁小徽标/关于 | 低频但排障需要 | 过度隐藏会影响截图识别 |
| 产品说明句 | Remove/Move | 帮助 tooltip | 无日常决策价值 | 新用户失去说明，需帮助入口补偿 |
| 在线/运行/阻塞 KPI | Keep once | Header | 首屏健康总览 | 必须定义唯一真源 |
| Dock 在线/运行/阻塞 | Merge | Header | 删除重复状态 | Dock 需保留数据新鲜度与全局反馈 |
| 最近更新时间 | Keep | 轻量状态栏 | 判断数据是否陈旧 | 需说明更新时间口径 |
| StatusStrip connector | Keep | Header | 连接器限制影响派活 | 不得把 disabled 误报为可用 |
| latest event | Keep once | Header 或右事件区 | 快速确认最近变化 | 需要“查看全部”入口 |
| 左轨最新消息 tile | Remove | - | 与 latest event 重复 | 无 |
| 右侧 event toast | Merge | 结果/日志入口 | 与当前 Agent 结果关联 | 需保留错误可见性 |
| 调度身份/派工状态 | Collapse | 左轨治理摘要 | 平时稳定，异常时展开 | 异常必须主动提示 |
| blocker 摘要 | Keep | 左轨顶部 | 高决策价值 | 默认只显示 2-3 行并可展开 |
| blocker 全文 | Collapse | 详情抽屉/展开区 | 保留治理证据 | 长文本需可复制 |
| 角色 35 / Lanes 30 | Collapse | 左轨二级折叠 | 低频治理信息 | 搜索/过滤比全量卡更重要 |
| 中央 Agent 名称/状态 | Keep | AgentCard | 矩阵扫描核心 | 无 |
| 中央 Agent 当前任务 | Keep | AgentCard | 判断工作负载 | 限 1-2 行 |
| 中央 5 个动作 | Collapse | 选中卡 2 个主动作 + 更多菜单 | 减少 8 卡按钮洪水 | 更多菜单需可键盘访问 |
| 右侧头像/名称/状态 | Merge | 紧凑详情头 | 保留上下文 | 不再重复 quote 气泡 |
| 右侧 quote | Collapse | 可选状态提示 | 趣味性高、决策价值低 | 运行异常时可显示有用告警 |
| 右侧 4 项指标 | Merge | 2 行紧凑摘要 | 给任务表单让位 | 详细指标放“详情”折叠 |
| 快捷任务 | Keep | 下发任务首屏 | 高频入口 | 最多 2-3 条，避免卡片化过度 |
| 自定义任务表单 | Keep | 下发任务首屏 | 核心动作 | CTA 必须首屏可达 |
| 任务队列/日志 | Keep | 右侧 tabs | 完成闭环 | tabs 顺序保持任务->队列->日志 |
| Corner Assist | Move/Conditional | 右轨收起或窄屏时显示 | 只在详情不可见时补位 | 避免与右轨并存 |
| Dock | Reduce | 底部轻量状态条 | 保留更新时间与全局反馈 | 不遮挡 corner assist |

## 7. 无用信息判定标准

一项信息只有同时满足以下至少一项，才允许删除或折叠：

1. 与其他区域表达同一事实，且没有更细粒度操作。
2. 在核心 5 个任务中不影响任何决策。
3. 长期稳定，只在排障或审计时使用。
4. 不可操作、无明确下一步，并已有更强真源。
5. 使核心动作退出首屏或显著延长焦点路径。

“文字多”“看起来复杂”“颜色不好看”不能单独构成删除理由。

## 8. 推荐改造顺序

### P0：核心路径重排

1. 右详情改为“紧凑 Agent 头 + tabs + 内容”，将 quote 与详细指标移入可展开详情。
2. 1280x720 下“任务名称、命令、立即派活”无需滚动即可见。
3. 去掉 Header/Dock/左轨的重复 KPI 和 latest event。
4. 中央非选中卡只保留选择与一个主动作；其他动作进入选中卡或菜单。

失败条件：核心信息被删失、CTA 仍需滚动、状态口径出现多个真源、Agent 快捷动作不可达。

### P1：渐进披露与键盘路径

1. 左轨改为“异常摘要 + 治理详情”，角色/Lanes 支持搜索或状态过滤。
2. 右轨收起时才显示 Corner Assist；右轨展开时隐藏触发器。
3. 建立跳转顺序：全局健康 -> Agent 矩阵 -> 右 tabs -> 左治理详情。
4. 图标按钮补充可见 tooltip、`aria-label` 和稳定焦点样式。

失败条件：异常被折叠后不可发现、焦点陷入隐藏区、搜索无结果反馈。

### P2：视觉系统收敛

1. 统一标题、正文、辅助、代码四级字号。
2. 收敛边框、阴影和状态饱和度；选中只使用一个主要强调层。
3. 静态品牌说明移入帮助入口。
4. reduced-motion 下关闭非必要浮动、脉冲和位移动画。

失败条件：状态只能靠颜色识别、hover 引发布局位移、文字在 125%/150% DPI 下互压。

## 9. 推荐信息架构

```text
┌──────────────────────────────────────────────────────────────┐
│ 品牌/版本 | 在线 8 | 运行 0 | 阻塞 1 | connector | 最近事件 │
├──────────────┬─────────────────────────────┬─────────────────┤
│ 异常与治理   │ Agent 工位矩阵              │ Operator        │
│              │                             │ [任务 队列 日志]│
│ blocker 摘要 │ [1][2][3][4]                │ Agent 紧凑头    │
│ [展开详情]   │ [5][6][7][8]                │ 快捷任务        │
│              │                             │ 任务名/命令     │
│ 角色/Lanes   │                             │ [立即派活]      │
│ [折叠/筛选]  │                             │                 │
├──────────────┴─────────────────────────────┴─────────────────┤
│ 数据更新时间 | 全局反馈                                      │
└──────────────────────────────────────────────────────────────┘
```

响应式策略：

- `1440x900+`：三列常驻，右轨优先保证任务 CTA。
- `1280x720`：三列保留但压缩只读详情；左轨可收起，任务表单首屏可见。
- `<1120`：中央矩阵优先；左右轨使用互斥抽屉，不允许两轨覆盖中央。
- 宽屏：增加中央留白或卡片宽度，不无限放大左右轨。

## 10. 视觉与交互规则

- 背景、surface、边框和文本至少形成 4 个明度层，不做一色到底。
- 状态必须包含文本或图标，不只使用红绿边框。
- 选中卡只使用“边框 + 轻阴影”一个主要强调，不使用缩放造成位移。
- 可点击区域最小 32x32；主 CTA 高度至少 36px。
- 滚动条只出现在明确容器；页面本身在桌面视口不产生水平滚动。
- 长 blocker、命令和端点支持截断、tooltip 与复制，不强行撑宽布局。
- 空态说明下一步；错误态包含原因、影响和可执行动作。
- `prefers-reduced-motion` 下移除非必要脉冲与位移。

## 11. 核心交互目标

正常路径：

1. 进入控制舱即可看到全局健康与 blocker。
2. 一次点击选择 Agent。
3. 一次点击切到任务 tab（默认已在任务 tab 时为 0 次）。
4. 快捷任务最多 1 次选择；自定义任务直接填写。
5. CTA 首屏可见并提交。
6. 提交后自动进入队列或显示明确结果入口。

目标：从进入控制舱到快捷任务提交不超过 3 次点击；键盘从页面入口到 Agent 矩阵、右侧 tabs、任务表单均有区域级跳转，不再顺序穿越全部 8 卡动作。

## 12. 可量化验收

- Header/Dock/左轨重复 KPI 从当前多处表达收敛为 1 个主表达 + 1 个数据新鲜度入口。
- latest event 主表达不超过 1 处；其他区域只提供入口或上下文关联结果。
- 1280x720 下任务名、命令和“立即派活”无需滚动可见。
- 1280x720、1440x900、宽屏均满足 `scrollWidth == clientWidth`，无控件重叠。
- 右轨展开时 Corner Assist 不重复出现；右轨收起时可用。
- 非选中 AgentCard 的直接动作数量减少，全部动作仍可在选中/菜单路径访问。
- 键盘到任务 tab 的焦点步骤显著低于当前约 75；实现阶段记录 before/after 数量。
- 文本对比度满足正常文本 4.5:1；状态有文本/图标双通道。
- 125%/150% DPI 与长中文/英文命令不互压。
- `orchestration:check/report/preflight/connector-safety`、`lint`、`build`、`git diff --check` 全通过。

## 16. 最小开工切片

第一张实现卡只做 P0：去重全局状态、重排右侧任务优先级、减少非选中卡动作、条件隐藏 Corner Assist。P1/P2 在 P0 通过 1280x720 与键盘路径验收后再开，不进行全页重写。
