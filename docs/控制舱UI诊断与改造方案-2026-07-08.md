# 桌面牧场控制舱 UI 诊断与改造方案 · 2026-07-08

> **状态**：草稿 v0.2（整理版，待拍板进入实施）
> **触发**：2026-07-08 用户反馈“当前布局混乱，信息显示错误”
> **定位**：C0-1~C0-6 accepted 之后的控制舱 UI 增量诊断，不回滚已验收基线
> **目标**：把截图中能直接读出的错误、冲突和噪声定位到数据源与 UI 区域，形成可拍板、可拆任务、可验收的改造方案
> **基线**：[`docs/主页面重构方案-v3.0-控制舱收口.md`](./主页面重构方案-v3.0-控制舱收口.md)

## 0. 结论先行

这次问题不是单个组件布局失控，而是四类问题叠加：

| 类别 | 主要表现 | 处理优先级 |
|---|---|---|
| 信息真源不一致 | 截图、`agentCore.ts`、`agentAnimals.ts`、`status.json` 对同一头牛马的命名和状态表达不一致 | P0 |
| 文案语义混用 | “打工人黑话”“赛马/牧场黑话”“工程任务 ID”在同一区域并列展示 | P0 |
| 汇总指标重复 | 顶部 KPI、StatusStrip、左侧 dispatch、底部 dock 都在汇报在线、任务、阻塞、最近更新 | P1 |
| P0 收口未贯彻到底 | 右侧 Tab 已做，但 Tab 外仍泄露任务负载；左侧和底部仍像数据 dump | P1 |

推荐处理顺序：

| 阶段 | 目标 | 预估 | 结论 |
|---|---|---:|---|
| P0 信息对齐 | 统一标题、角标、codename、字段标签和 0 值说明 | 2h | 先做 |
| P1 视觉降噪 | 收敛顶部、左侧、右侧、底部的信息重复 | 4h | 先做 |
| P2 主次强化 | 中央 8 卡选中态、hover 态、配色饱和度收敛 | 2h | 跟进 |
| P3 品牌拍板 | 8 个 codename 终稿由用户确认后写入 | 待定 | 不抢跑 |

## 2. 问题盘点

### 2.1 字面错误和命名错位

| # | 现象 | 真源/位置 | 判断 |
|---|---|---|---|
| 1 | “控制台”和“控制舱”混用 | `NiuMaWorkspace.tsx`、`electron/main.ts`、`ranch.html` | 统一为“桌面牧场 · 控制舱” |
| 2 | 副标题 `OPS-RANCH v2.6` 与窗口标题语义不一致 | `NiuMaWorkspace.tsx` header | 保留为版本/代号，不承担标题含义 |
| 3 | “牛马工作位阵”/“牛马工位矩阵”口径不统一 | 中央区标题 | 推荐统一为“牛马工位矩阵” |
| 4 | `[1号角]` / `[1号舍]` / 牛马名重复表达同一件事 | `agentCore.ts`、`agentAnimals.ts`、中央卡 | 推荐删除角标；如必须保留，改为 `[1号位]` |
| 5 | 8 个 codename 截图态和代码态不一致 | `agentCore.ts`、`agentAnimals.ts` | 不能把截图态当当前真源；需用户拍板后同步 |
| 6 | `RANCH v2.5 / IDE / QA / UI` 等徽章与“牛马 Cow”不在同一语义层 | `AGENT_SEEDS.badge` | badge 应降级为角色/能力标签，不与品牌名并列 |
| 7 | “温温 31°C” | 右侧详情 | 改为“温度” |
| 8 | `0% / 100%` 无 label | 右侧详情 meter | 明确为“压力 0% / 电量 100%”或对应业务字段 |
| 9 | `R0-2 deploy visible probe` 直接暴露 | 右侧任务详情 | 主文案用可读任务名，ID 作为 secondary metadata |

### 2.2 数值和状态互相打架

| # | 冲突 | 用户会怎么误读 | 处理方向 |
|---|---|---|---|
| 1 | 顶部“正在拉磨 0” vs 底部“在线牛马 8 头” | 在线但全都没干活，像异常 | 顶部区分“在线 8 / 在跑 0 / 阻塞 3” |
| 2 | “Gate 3 blocked” + “connector · 3 blocked” + “tasks 0/11” | 不知道 3 和 11 分别是哪一层 | 阻塞只在全局健康区主显一次 |
| 3 | 右侧“任务负载 0” vs 底部“历史功绩 11” | 右侧像丢数据 | 明确右侧是“当前选中牛马”，底部是“全局历史” |
| 4 | “活跃 Lanes 2 / 1 阻塞 / 10 待命”与左侧“今日阻塞 1” | 两个 1 的统计范围不明 | lane 状态归编排区，今日新增归 dispatch 区 |
| 5 | “已打开任务”与“最近更新”挤在一起 | 任务状态和时间戳混成一组 | 时间戳统一放 statusbar 或详情 metadata |

### 2.3 布局噪声

| 区域 | 问题 | 目标 |
|---|---|---|
| 顶部 | KPI、StatusStrip、设置、重置重复并列，汇总指标过多 | 只保留 3 个全局健康指标和一个设置入口 |
| 左侧 | blocker 全文、current target 全文、role/lane 卡片大量展开 | 默认摘要化，长文折叠，role/lane accordion |
| 中央 8 卡 | 角标、名称、badge、codename、emoji、按钮、meter、状态名同时出现 | 保留身份、状态、少量操作；选中卡主次明显 |
| 右侧 | Tab 已存在，但任务负载/任务数/派活按钮仍漏在 Tab 外 | Tab 外只放身份和运行指标，任务内容全部进 Tab |
| 底部 dock | 6 个数据块横排，重复顶部和左侧信息 | 改为一条 compact statusbar |
| 配色 | 8 卡 accent 都高饱和，选中态不够明显 | 选中卡更亮，非选中卡降饱和但保持可读 |

## 3. 根因分析

### 3.1 一牛三源

| 数据 | 文件 | 当前职责 | 问题 |
|---|---|---|---|
| `AGENT_SEEDS` | `src/lib/agentCore.ts` | 控制舱 agent 身份、badge、codename | 应该成为身份文案主真源 |
| `AGENT_ANIMALS` | `src/ranch/data/agentAnimals.ts` | 桌面牧场浮窗 animal 展示 | 当前有独立文案，容易漂移 |
| `roles[]` / `lanes[]` | `docs/orchestration/status.json` | 编排状态、任务、角色 | 不应覆盖品牌/codename 真源 |

结论：`AGENT_SEEDS` 应作为牛马身份文案单一真源；`AGENT_ANIMALS` 应派生或强同步；`status.json` 只保留编排状态和引用关系。

### 3.2 文案风格分裂

| 风格 | 例子 | 适合放哪里 |
|---|---|---|
| 打工人黑话 | 批发排障、带薪拉屎、废话周会、打水摸鱼 | 可以保留在历史/调侃层，但不适合做主品牌身份 |
| 赛马/牧场黑话 | 远咒马牌、稳赢马牌、跑法回合、打水赶鱼 | 适合做桌面牧场角色 codename，但需统一终稿 |
| 工程任务 ID | R0-2、deploy visible probe、connector gate | 只做 metadata，不做主标题 |
| 状态剧本 | 熟睡躺平、疯狂码砖、生产救火 | 属于 `STATE_METAS`，本次不改 |

结论：agent codename 和 state name 必须分层。codename 是“这头牛马是谁”，state 是“它此刻在干什么”。

### 3.3 汇总信息没有唯一展示位

同一组全局数字在顶部 KPI、StatusStrip、左侧 dispatch、底部 dock 四处出现。用户会自然选择最显眼的一个当真源，但四处口径不同，导致“信息显示错误”的感受。

治理原则：同源数据只主显一次，其他位置只能引用、摘要或作为 drill-down。

## 4. 改造方案

### 4.1 信息真源统一

| 项 | 方案 | 涉及文件 |
|---|---|---|
| agent 身份文案 | `AGENT_SEEDS` 作为唯一主真源 | `src/lib/agentCore.ts` |
| 桌面牧场 animal 展示 | 从 `AGENT_SEEDS` 映射 `displayName`、`codename`、`glyph`、`defaultPosition` | `src/ranch/data/agentAnimals.ts` |
| 编排状态 | `status.json` 只保存 role/lane/status，不直接维护 codename | `docs/orchestration/status.json` |
| UI 展示 | 组件按“全局 / 当前选中 / 历史 / metadata”分类渲染 | `NiuMaWorkspace.tsx`、`StatusStrip.tsx` |

### 4.2 文案修正

| 项 | 默认建议 | 备注 |
|---|---|---|
| 产品名 | 桌面牧场 · 控制舱 | 统一窗口、header、文档 |
| 中央区标题 | 牛马工位矩阵 | 比“工作位阵”更稳 |
| 角标 | 删除 | 如保留则用 `[1号位]` |
| 任务数 | 当前任务 / 历史功绩分开 | 避免 0 vs 11 打架 |
| meter | 压力 / 电量 / 温度明确 label | 不再裸露百分比 |
| 任务 ID | 放 secondary metadata | 用户先看可读任务名 |

### 4.3 页面降噪

| 区域 | 调整 |
|---|---|
| 顶部 | KPI 收敛为“在线 / 在跑 / 阻塞”三个核心指标，StatusStrip 删除 `tasks 0/11` 主显 |
| 左侧 | blocker、current target 默认折叠；role 和 lane 改 accordion，只展开关键 4-6 项 |
| 中央 | 删除或降级 slot 角标；选中卡 scale + accent + shadow，非选中卡降权 |
| 右侧 | 任务负载、任务数、立即派活全部归入对应 Tab；Tab 外只保留身份和运行指标 |
| 底部 | 6 个 dock 数据块折为一条 statusbar：在线、在跑、阻塞、最近更新、设置入口 |

## 7. 验收标准

### 7.1 自动验证

```bash
npm.cmd run lint
npm.cmd run build
npm.cmd run orchestration:check
```

如本次触及 connector 相关状态，还需要补跑：

```bash
npm.cmd run orchestration:preflight
npm.cmd run orchestration:connector-safety
```

### 7.2 视觉验收

| 项 | 验收口径 |
|---|---|
| 信息一致性 | 同一 agent 的名称、codename、badge 在控制舱和桌面牧场中一致 |
| 层级 | 顶部只读全局健康，右侧只读选中牛马，底部只读全局状态摘要 |
| 空态 | 当前任务为 0 时有明确说明，不与历史功绩混淆 |
| 选中态 | 中央 8 卡中当前选中卡有明显主视觉 |
| 响应式 | 1366×768 和 1920×1080 下无文字重叠、按钮溢出、卡片错位 |
| 卖点保护 | `NiuMaAvatar.tsx` 和 `STATE_METAS` 不因本次改造发生结构性变化 |
