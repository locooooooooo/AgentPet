# 牛马 Hub 项目定位 v1.0

> 日期：2026-07-17
>
> 产品名：牛马 Hub（NiuMa Hub）
>
> 项目代号：`multi-agent-niuma`
>
> 定位状态：项目级北极星；不等同于任何 Connector 已通过执行验收

## 1. 定位结论

牛马 Hub 是一个**本地优先的多 Agent 安装、运行与协作中枢**。用户可以在一个桌面应用里发现、安装、启动和管理不同厂商或开源 Agent，并通过统一任务、Session、日志、权限和调度能力，让多个 Agent 真实协作。

牛马场继续保留，而且是产品最鲜明的识别符：它不再代表整个产品品类，而是 Hub 的**标志性实时状态空间**。每只牛马对应一个真实 Agent 或 Agent 实例，其动作、表情、台词和音效必须来自真实运行状态，不得用装饰动画伪造执行结果。

一句话对外表达：

> **把散落在电脑里的 Agent 牵进同一个牛马场，统一安装、开工、协作和看状态。**

## 2. 产品身份

| 项目 | 决策 |
| --- | --- |
| 对外名称 | 牛马 Hub |
| 英文名称 | NiuMa Hub |
| 产品品类 | 多 Agent 桌面 Hub / 本地 Agent 协作中枢 |
| 核心能力 | Agent 生命周期管理 + 多 Agent 协调 + 真实状态可视化 |
| 特色体验 | 牛马场、牛马黑话、身份动物、状态动作、可定制皮肤与音效 |
| 首发平台 | Windows 桌面端；延续现有 Electron 技术基线 |
| 默认原则 | 本地优先、显式授权、真实状态、可审计、可恢复 |

牛马 Hub 不是“桌面牧场加几个 Agent 图标”，也不是“把多个聊天窗口塞进一个壳”。它的产品价值必须由完整闭环构成：

```text
发现/安装 Agent
  -> 检测与配置
  -> 启动 Agent 或建立 Session
  -> 下发单 Agent / 多 Agent 任务
  -> 调度依赖、并发、取消和重试
  -> 聚合日志、产物、权限和终态
  -> 牛马场与音效实时回执
```

## 3. 目标用户与核心任务

### 3.1 目标用户

- 同时使用 Codex、Trae、Qoder、OpenCode、OpenClaw 等多个 Agent 工具的个人开发者。
- 需要比较不同 Agent 能力、把任务交给合适 Agent、追踪多个任务状态的重度用户。
- 希望保留本地数据与执行控制，不愿把所有项目和凭据交给单一云端平台的用户。
- 喜欢个性化桌面体验，但仍要求工具可靠、安静、可恢复的用户。

### 3.2 用户要完成的五件事

1. **收进来**：知道这台电脑有哪些 Agent、缺哪些、从哪里安全安装。
2. **开起来**：从 Hub 启动、聚焦、停止或重新连接 Agent，不在桌面上找入口。
3. **管起来**：统一查看版本、认证、权限、工作目录、Session、日志和更新状态。
4. **协作起来**：把一个目标拆给多个 Agent，处理依赖、并发、失败、取消和结果交接。
5. **养起来**：用牛马场、皮肤和音效建立自己的工作氛围，同时一眼读懂真实状态。

## 4. 产品支柱

### 4.1 Agent Library：发现、安装与启动

Hub 提供统一 Agent 库。每个 Agent 由版本化清单描述，至少包含官方来源、平台、发现方式、安装方式、启动方式、更新方式、能力等级和风险提示。

“支持某 Agent”必须拆成六个独立等级，UI 不得只显示一个含糊的“已支持”：

| 等级 | 含义 |
| --- | --- |
| `catalogued` | Hub 认识该 Agent，并有可信来源和说明 |
| `detected` | 已在当前机器发现安装或可执行入口 |
| `installed` | Hub 已验证安装结果和版本 |
| `launchable` | Hub 可以安全启动或聚焦它 |
| `connectable` | Hub 能读取真实 Session、状态、日志或结构化输出 |
| `coordinatable` | Hub 能受控下发任务、取消、超时并取得可审计终态 |

安装有三种正式路径：

- **包管理器安装**：优先使用 `winget`、官方安装器或 Agent 官方推荐的包管理器。
- **官方引导安装**：Hub 打开经过校验的官方页面，并在安装后重新检测。
- **已有安装接入**：用户选择或 Hub 发现现有可执行文件，再做身份、版本和能力校验。

任何下载安装都必须先展示来源、发布者、版本、权限、命令和目标目录。需要管理员权限、修改 `PATH`、写入 shell 配置或安装后台服务时，必须单独确认。Hub 不从未知镜像静默下载，不把“下载完成”当成“安装成功”。

### 4.2 Agent Runtime：统一会话与真实状态

不同 Agent 通过独立 Adapter 接入统一 Runtime。Adapter 负责能力声明、进程或 API 生命周期、输入输出转换、错误归一化和健康检查；Runtime 负责统一 Session、任务、事件、日志与审计。

接入形态分为：

- **Headless/CLI Agent**：可在 Hub 内直接执行、流式读取、取消和协调，是多 Agent 编排主路径。
- **Desktop GUI Agent**：首期提供检测、安装、启动、聚焦和健康状态；只有存在稳定且获准的本地 API/CLI 时才升级为可协调。
- **Web Agent**：先提供可信深链和账号状态提示；不以 iframe 冒充原生接入。

“全部放入 Hub”在产品上指统一入口、统一状态和统一协作面。受操作系统、安全策略或第三方能力限制的 GUI 不承诺强行嵌入 Hub 窗口。

### 4.3 Orchestration：多 Agent 协调核心

牛马 Hub 的核心不是启动器，而是现有控制舱、Connector Runtime 和本地调度能力的产品化：

- 按 Agent 能力、模型、成本、权限或用户指定进行任务分派。
- 支持 `dependsOn`、串并行、单 Agent 互斥、全局并发上限、超时、取消和受控重试。
- 上游失败时明确阻断或请求人工决策，不静默跳过。
- 每次状态转换产生不可变、脱敏的审计事件。
- 任务结果可交给下一个 Agent，保留来源、文件变更、验证和失败上下文。
- UI、牛马场、通知和音效只消费同一份 Runtime 真值，不各自推断终态。

### 4.4 Ranch：牛马场特色状态空间

牛马场承担三个职责：

1. **状态总览**：用动作和表情表达 `idle / queued / running / waiting / success / error / offline`。
2. **快速操作**：选中牛马后可查看任务、日志、Session、权限请求，或跳转控制舱。
3. **情绪价值**：身份动物、黑话台词、个性动作和环境反馈让多 Agent 运转具有记忆点。

硬边界：没有 Runtime 证据时只能显示“未知、离线或未接入”，不得播放“完成”动作或成功音效。装饰性的摸鱼、散步和环境动画不得改变业务状态。

### 4.5 Personalization：Hub 皮肤与音效

个性化不是散落的 CSS 开关，而是两个版本化内容系统：`HubTheme` 与 `HubSoundPack`。

#### Hub 原生皮肤

Hub 自己的皮肤使用原生主题包，不需要注入自身页面。主题包至少包含：

- `manifest.json`：主题 ID、名称、作者、版本、兼容范围和资产许可。
- 颜色与排版 Token：背景、表面、文字、边框、状态色、密度和字体策略。
- 背景与装饰资产：纯背景、纹理、牛马场环境、可选动物外观。
- 模式配置：浅色、暗色、高对比、减少动态、背景焦点与安全区。
- 完整性信息：文件哈希、包大小、最低 Hub 版本和迁移策略。

主题必须支持预览、应用、切换、导入、导出、恢复默认和坏包回退。主题只能改变表现层，不能携带可执行脚本、修改 Connector 权限或影响任务真值。

#### 对 Codex-Dream-Skin 的借鉴边界

2026-07-17 核对的 [Fei-Away/Codex-Dream-Skin](https://github.com/Fei-Away/Codex-Dream-Skin)（当时检视提交 `a1c48b3`）采用本机回环 CDP，将主题配置、纯背景、CSS 和 renderer 注入组合起来；它强调不修改官方安装包、`app.asar` 或签名，并提供身份校验、验证、暂停与恢复路径。

牛马 Hub 借鉴以下方法：

- 主题配置与素材分离，主题可保存和切换。
- 根据图片明暗、焦点和安全区适配真实控件。
- 应用前校验，应用后做可交互与可读性验证。
- 所有改造可暂停、可恢复，失败时保留原配置。
- 本机调试入口仅绑定回环地址并校验目标身份。

但 Hub 自身不需要 CDP 注入。未来若提供“外部应用皮肤桥”，必须按 Agent 单独实现、默认关闭、明确标为实验能力，并在第三方应用更新后重新验收。不得修改第三方二进制、绕过签名或把脆弱 DOM 选择器当成稳定平台 API。

参考仓库的根 README 指向 `macos/LICENSE`（MIT）；实际复制代码或分发素材前，仍需核对许可覆盖范围、NOTICE、人物/IP/商标和每项资产来源。现阶段只确认设计方法，不把其示例素材纳入牛马 Hub。

#### Hub 音效

音效采用“事件语义音 + Agent 身份尾音 + 音效包”的三层模型，并承接现有《牛马状态回执音效规范 v0.1》：

- 事件语义音表达开始、等待、完成、失败、需确认、断线等真实状态。
- Agent 身份尾音用于区分是哪只牛马，不覆盖事件主语义。
- 音效包提供整体风格，可自定义、导入、预览、切换和恢复默认。
- 支持全局音量、每事件开关、每 Agent 开关、安静时段、频率限制和“仅错误/需确认”。
- 多窗口与多视图必须去重，同一个业务事件最多播放一次。
- 自定义音频只本地读取；导出或分发音效包前必须记录授权与来源。

## 5. 信息架构

首期 Hub 使用六个一级入口：

| 入口 | 用户问题 |
| --- | --- |
| 总览 | 现在有哪些 Agent 在工作，有没有阻塞或需要我处理？ |
| Agent 库 | 有哪些 Agent，是否安装，能做到哪一级接入？ |
| 工作台 | 如何创建、编排、取消和复盘多 Agent 任务？ |
| 牛马场 | 所有 Agent 的实时状态是什么？ |
| Session / 日志 | 某次执行发生了什么，产物和错误在哪里？ |
| 个性化与设置 | 如何切换皮肤、音效、权限、安装源和通知策略？ |

控制舱的 DockView、多实例和多窗口能力继续服务“工作台”，但布局真值与业务真值必须分离：关闭面板不停止 Agent，复制面板不复制任务，多窗口不重复通知或响铃。

## 6. 产品原则

1. **协调价值高于入口数量**：图标再多也不等于多 Agent Hub；至少两个真实 Adapter 能完成可审计协作，才形成核心价值。
2. **真实状态高于视觉热闹**：牛马动作、徽标、通知和音效都必须来自统一真值。
3. **分级支持高于“万能兼容”**：清楚标出每个 Agent 到了哪一级，能力不足就停在启动或深链层。
4. **本地优先但不假装离线**：本地保存编排与审计；Agent 自身需要云服务时如实展示网络和账号依赖。
5. **显式授权高于自动化冲动**：安装、提权、写文件、启动外部进程和发送任务都有可见边界。
6. **可恢复高于一次成功**：安装、主题、音效、布局与 Connector 配置都必须有回滚或默认恢复路径。
7. **个性化服务于识别与情绪**：皮肤和音效可以强烈，但不能降低可读性、掩盖错误或干扰连续工作。

## 7. 首期范围与非目标

### 7.1 首期范围

- Windows 本地 Agent 库、可信安装计划、安装检测、版本与更新状态。
- 至少 3 个 Agent 达到 `launchable`，其中至少 2 个 Headless Agent 达到 `coordinatable`。
- 单一 Runtime 真值下的 Session、任务、依赖、并发、取消、超时、重试、日志和审计。
- 控制舱作为详细工作面，牛马场作为标志性实时总览。
- Hub 原生主题包、音效包、默认恢复和防打扰设置。
- 所有 Agent 清楚展示接入等级、缺失条件和下一步，而不是统一显示“在线”。

### 7.2 首期非目标

- 不承诺把所有第三方 GUI 强制嵌入一个 Electron 窗口。
- 不做无审核的通用 Agent 应用商店或任意脚本市场。
- 不代替第三方账号、订阅、模型额度和许可证管理。
- 不静默提权、安装服务、改 `PATH`、写凭据或开启外部执行。
- 不修改第三方官方二进制、签名、`asar` 或受保护安装目录。
- 不把外部应用换肤作为多 Agent 协作成立的前置条件。
- 首期不做团队云端协作、远程算力编排或移动端控制。

## 8. MVP 验收定义

只有同时满足以下条件，产品才可以称为“牛马 Hub MVP”：

| 编号 | 验收结果 |
| --- | --- |
| HUB-01 | Agent 库展示至少 3 个经来源校验的 Agent，并分别显示六级支持状态 |
| HUB-02 | 至少 3 个 Agent 可完成检测、安装或官方引导、启动/聚焦与重新检测闭环 |
| HUB-03 | 至少 2 个 Headless Agent 通过各自 Adapter 完成真实结构化任务、取消、超时和终态采集 |
| HUB-04 | 一个包含依赖关系的双 Agent 工作流真实完成，失败时下游不会误启动 |
| HUB-05 | 控制舱、牛马场、通知和音效对同一任务显示一致终态，且业务事件不重复 |
| HUB-06 | 权限、cwd、环境变量、安装来源、命令和审计记录可查看且脱敏 |
| HUB-07 | 至少 2 套 Hub 原生主题可切换并恢复默认，坏主题包不会阻止启动 |
| HUB-08 | 至少 2 套音效包可切换，支持静音、仅重要事件和安静时段 |
| HUB-09 | Hub 重启后可以恢复 Agent 清单、布局和历史 Session，但不会擅自重启任务 |
| HUB-10 | 外部 Agent 不可用、未认证或版本不兼容时明确降级，不用模拟数据冒充成功 |

## 9. 演进路线

### R0：定位与合同

- 固化本定位、术语、支持等级和 MVP 验收。
- 将现有 `ConnectorConfig`、`AgentInstance`、scheduler、牛马状态和通知能力映射到 Hub 模型。
- 定义 `AgentManifest`、`InstallPlan`、`HubTheme` 和 `HubSoundPack` 的版本化合同。

### R1：Agent Library 与 Lifecycle

- 建立 Agent 库、机器发现、版本检测、可信来源和安装计划。
- 打通安装确认、进度、失败、回滚提示、启动、聚焦和更新状态。
- 首批 Agent 只按证据授予支持等级，不预设全部可协调。

### R2：多 Agent 协作闭环

- 选择两个确有 Headless 接口的 Agent 完成 Adapter 验收。
- 用现有本地 scheduler 完成依赖、并发、取消、超时、重试和审计闭环。
- 将真实 Session/Task/AgentInstance 投影到控制舱和牛马场。

### R3：个性化系统

- 上线原生主题包、音效包、导入导出、预览、恢复与兼容检查。
- 建立官方默认包与本地自定义包隔离，补足素材许可元数据。
- 评估 Codex 外部皮肤桥 Beta，但不与 Hub 原生主题系统耦合。

### R4：生态扩展

- 提供有权限清单、签名、兼容声明和审核流程的 Adapter SDK。
- 评估主题/音效/Adapter 分享中心；默认不执行第三方任意脚本。
- 在 Windows 闭环稳定后评估 macOS 与 Linux。

## 10. 与现有项目资产的关系

| 现有资产 | 新定位下的角色 |
| --- | --- |
| Electron 主进程与 preload | Hub 本地能力、窗口、安装/启动与受控 IPC 边界 |
| `connectorGate` / `connectorRuntime` | Adapter 执行与权限安全底座 |
| `AgentInstance` 投影与本地 scheduler | 多 Agent 真值与协调核心 |
| 实时 Agent 控制舱 | Hub 工作台与深度操作面 |
| 桌面牧场 / `RanchApp` | 标志性实时状态空间，不再单独承担整个产品定位 |
| DockView / 多窗口需求 | Hub 可定制工作面与多屏效率能力 |
| 状态回执音效规范 | `HubSoundPack` 的事件语义与防打扰基线 |
| 现有首页 | 后续改为 Hub 总览，不再以“牧场落地页”作为主要叙事 |

## 11. 当前事实与立即决策

截至 2026-07-17，仓库已经具备 Electron 壳、控制舱、牛马场、Connector 安全门、Codex/Trae/Qoder 接入策略、Session/AgentInstance 真值和本地 scheduler 基础；但 Agent 自动下载安装、通用 Agent Library、原生主题包和可导入音效包尚未形成产品闭环。

当前 Codex 与 Trae 仍是 `draft / pending / disabled`，Qoder 是 `disabled / rejected`。新定位不会自动抬高它们的支持等级，也不构成开启外部执行的授权。

下一轮需求设计应只补四份基础合同：

1. `AgentManifest + InstallPlan`：Agent 库、来源、发现、安装、版本和支持等级。
2. `Adapter Capability`：每个 Agent 可启动、可连接、可协调能力与降级规则。
3. `HubTheme`：原生皮肤包、预览、兼容、恢复和资产许可。
4. `HubSoundPack`：事件语义、身份尾音、防打扰、去重和本地资产合同。

这四份合同冻结后，再拆独立实现任务；不在一个大任务里同时改安装器、Runtime、控制舱、牛马场、主题和音效。
