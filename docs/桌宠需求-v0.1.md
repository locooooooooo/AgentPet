## 一、缘起：为什么是"桌宠"

### 1.1 当前形态的真正瓶颈

我们现有一个完整可跑的多 Agent 控制舱：
- 1220×860 单窗口，8 张 agent 卡 + 一个右侧详情面板
- 任务下发 → 模拟 runner 推进 → 任务队列 + 流式日志
- 完整 orchestration 状态面板（PM / Supervisor / 短工 / lanes / connectors）
- 本地 runner 可 spawn 真实子进程
- 仿真数据全部从 Electron `userData/agent-data/agents.json` 拉

**这形态对"操作员"是好的，对"长时间不在电脑前"的开发者是坏的：**
- 想看 agent 状态，得**手动呼出**控制舱窗口
- 任务完成 / 出错，没有桌面级通知
- 多个 agent 同时作业，**状态是割裂的**（要逐张卡去看）
- 缺乏"陪伴感" —— 它是个工程工具，不是个伙伴

### 1.2 桌宠是不是一个"伪需求"？

需要诚实回答：在中国开发者语境下，"桌宠"经常被默认成"二次元老婆""养成系小动物"——这一类桌宠 90% 是**情绪填补工具**，对实际工作流贡献很薄。

但我们要的桌宠不是这一类。我们的语境是：

> "让多 agent 系统的**状态流**变成一只看得见、会汇报、有性格的小生灵，长期待在桌面上作为 watchdog + 陪伴 + 工作看板。"

所以我们做的是 **"agent 状态拟人化看板 + 拟人化通知终端"**，不是"养成型盲盒"。这两个本质不同：

| 维度 | 养成型桌宠 | 我们要的桌宠 |
| --- | --- | --- |
| 核心价值 | 情绪填补 / 解压 | 状态可视 + 实时通知 + 偶尔的陪伴感 |
| 数据来源 | 游戏内事件 / 抽奖 | **真实 agent runtime** |
| 互动方式 | 喂食 / 摸头 / 换装 | 投喂任务、追问状态、订阅 lane |
| 期望寿命 | 长期玩下去 | 工作日 8h，陪伴但**不抢戏** |
| 失败模式 | 用户腻了卸载 | 用户**忘了它在** —— 说明它成功了 |

制作人立场：**这是工程产品的"软装"，不是独立游戏。** 它的成功标准是"安静地工作、适时地闹"，而不是"让用户沉迷"。

### 1.3 战略价值

如果这个方向成了，会产生三个东西：

1. **个人维度**：把现有控制舱从"工具"升级为"有个性的小生态"，开发者自己每天看着有满足感
2. **多 agent 操作员维度**：将来这套桌宠范式能**打包给其他多 agent operator**（任何跑 Codex/Trae/Qoder 多 agent 流水线的人都能拿去改一改用），从"个人项目"变成"产品雏形"
3. **演示维度**：多 agent 工作流的"门面担当" —— 截屏分享时它能直接体现"我的桌面有个多 agent 调度中心在跑"

## 二、目标用户画像

### 2.1 一级用户：你本人（个人开发者 / 多 agent operator）

- **典型场景**：你在公司用 Trae 写代码，让 Codex 跑测试，让 Qoder 做审计；你需要它们**别打扰你**，但**确实跑完了/出错了要马上知道**
- **痛点**：任务并行多了，状态在 IDE 各自终端里散落；等你想起来切过去看，已经过去一小时
- **期望**：桌宠一直飘在屏幕一角，安安静静；一旦有事，它会**主动报，但不过度报**

### 2.2 二级用户：你的同事 / 朋友（屏幕旁观者）

- **典型场景**：你的同事瞥到你桌面，看见一只小生灵在状态变化，会问"这是什么" —— 那是一个**30 秒电梯演讲**的机会
- **期望**：桌宠的视觉与状态变化要**自解释**到不需要讲解也能看懂

### 2.3 三级用户：将来的潜在用户（其他多 agent operator）

- **典型场景**：另一个跑多 agent pipeline 的开发者，搜到你的 repo，想 clone 改成自己的
- **期望**：代码要能改 `agents.json` 就能换"形象" / 改文案就能换"性格"，**不能锁死在我们的 8 个 agent 上**

## 三、核心使用场景（写下来不是装饰，是验收起点）

### 场景 A · 长跑任务的"安静的 watchdog"
> 你上班打开电脑，桌宠已经在屏幕右下角静静飘着，状态 idle（睡）。你下发 3 个长任务，然后切去 IDE 写代码 —— 中间 2 小时你不看桌宠。任务完成一只，桌宠小声呼吸一下、弹气泡「codex：构建失败一次回滚了」；任务全完成时它悄悄吹一个气泡「都下班了」。

**核心交付物**：状态变更 → 通知 → 自动收回，**不打断**

### 场景 B · 一眼可读的多 agent 看板
> 写代码期间瞥一眼屏幕右下：8 只 mini pet 一行，codex 在冒小齿轮，openclaw 在冒烟，hermes 在打盹。一眼就有了信息密度。

**核心交付物**：8 mini 集群 + 状态色点 + 状态名缩写

### 场景 C · 故障的情感化呈现
> 凌晨 2 点，一个任务栈崩了。桌宠**不像 sentinel 那样刷一堆红**，而是抖一抖、变红、冒一个气泡「hold on」；点开右侧控制舱能看到完整日志。

**核心交付物**：拟人化报警 + 控制舱联动（不是替代）

### 场景 D · 闲置时的桌面一抹"活物"
> 没任务时，桌宠偶尔眨眼、微微呼吸、偶尔换个姿势。它**不抢你的注意力**，但偶尔抬头看它会觉得"哦还在"。

**核心交付物**：idle 动效循环 + 鼠标 hover 才有的细节反应

### 场景 E · 想操作时一键召唤控制舱
> 你想看任务详情。右键桌宠，"召唤控制舱" → 控制舱从托盘弹出。你操作完，关掉它再回桌面，桌宠没动。

**核心交付物**：右键菜单 + 控制舱主窗口独立唤起

## 四、产品形态决策（每个决策背后都有备选）

### D1. 主宠 + 8 mini 集群（推荐）vs 一只大宠 vs 8 只平权大宠

**推荐**：1 主宠（大，拟人化，看一眼就懂）+ 8 mini pet（小，状态点，并排，作为"班表"）

理由：
- 单只拟人化最有"宠物感"，桌宠的"灵魂"靠它
- 8 个 agent 状态需要"一眼可读"，mini 集群是"班表"
- 8 只平权大宠会变成"桌面杂货摊"，体验差

### D2. 主宠形象 = 当前主控 agent（推荐）vs 主宠 = 永久固定代理

**推荐**：主宠显示**当前主控 agent**（默认 codex）；右键切换

理由：
- 多 agent operator 最关心"主控是哪个"，可视
- 永久固定一只主宠，剩下的就退化成"壁纸"，会腻
- 切主控 agent = 一次"自我重塑"，制造新鲜感

### D3. 桌宠窗口位置固定 vs 可拖

**推荐**：可拖（M1 加）+ 自动靠边（M2 加）

理由：
- 用户有自主感，自定义桌面布局
- 靠边自动吸附 = 不挡事

### D4. 桌宠 = 默认前置 vs 默认后台

**推荐**：alwaysOnTop + 用户可调两层（"漂浮" / "栖息"）

理由：
- 漂浮（默认）：看得见但不让它抢焦点
- 栖息（M1 加）：透明区穿透，桌面其他窗口能压住它，更"桌面共生"

### D5. 与控制舱 = 双形态共存（推荐）vs 单桌宠替代控制舱

**推荐**：双形态共存，桌宠**召唤控制舱**而**不替代**

理由：
- 控制舱是"精管"，桌宠是"陪伴 + 看板"，两个心智模型不同
- 强行合一会让控制舱失去"精管"的清晰度，桌宠也失去"陪伴"的纯净度

### D6. 状态通知 = 多通道（推荐）vs 仅气泡

**推荐**：气泡（不抢）+ 系统 Notification（关电脑也收到）+ 控制舱右上红点（精管回查）

理由：
- 单通道会被"自动隐藏""勿扰模式""窗口遮挡"各种场景截胡
- 多通道彼此冗余，**不过度报**比"绝对报"更重要

### D7. 与真实 agent 的耦合度（推荐：拆）

**推荐**：桌宠与真实 agent 完全解耦；它只订阅 `AgentSnapshot` 这一个 schema

- 即便没有真实 agent 接入，桌宠跑 simulation 也能"看上去跑得很真"
- 真 agent 接入只需要给 `agents.json` / IPC 推数据，桌宠不用改一行

## 五、核心体验地图（UX 关键时刻表）

下表是"剧本"，用来校验设计是否达到每个关键时刻。

| 阶段 | 用户动作 | 桌宠反应 | 体验目标 |
| --- | --- | --- | --- |
| 开机 | 启动应用 | 主宠从屏幕右侧滑入，8 mini 同时淡入，气泡「上工了 ~」 | "被欢迎" |
| 下任务 | 在控制舱点「立即下发」 | 主宠表情变为 coding；对应 mini 头顶冒齿轮；气泡「码起来了」 | "事情被看见" |
| 任务完成 | 任务进度到 100% | 主宠眨眼、张嘴笑；对应 mini 蹦一下；系统通知 + 气泡「完工」 | "被庆贺" |
| 任务失败 | 任务 exit code 非零 | 主宠颜色变红、抖一抖；对应 mini 冒烟；气泡「炸了，但能救」 | "被理解" |
| 闲置 | 5 分钟无任务 | 主宠呼吸放慢，眼睑渐垂，8 mini 各自打盹 | "不抢戏" |
| 鼠标 hover | 鼠标移上主宠 | 抬头看你一下，眼睛睁大，气泡「嗯嗯？」0.5s 后收回 | "被关心" |
| 右键 | 右键桌宠 | 弹出菜单：召唤控制舱 / 切主控 / 投喂 / 咖啡 / 摸鱼 / 戳 / 设置 | "被伺候" |
| 凌晨加班 | 持续低能自动状态 | 主宠困得睁不开眼、头上一弯月、偶发「修仙加班」气泡 | "被陪伴" |

## 六、8 个 agent 的"人设"（叙事层，不是文案）

如果桌宠要"有灵魂"，那 8 个 agent 不能只是 8 个 emoji，必须每个有**一致的角色动机**。这影响后续拟人化是否成立：

| Agent | 一句话人设 | 情绪底色 | 推荐形象（emoji 后续可换 SVG） |
| --- | --- | --- | --- |
| codex | 主力码农，扛活但偶尔自嘲 | 苦逼里带点黑色幽默 | 🐂 牛 |
| trae | 重构狂魔，对重复代码 0 容忍 | 强迫症 | 🐎 马 |
| qoder | 测试审计守门员，谨慎 | 佛系 | 🐂 |
| minimax | UI 设计师，审美洁癖 | 细节控 | 🐎 |
| workbuddy | 助理 / 日程，打水摸鱼 | 慵懒 | 🐂 |
| openclaw | 救火队长，事故响应 | 焦虑 + 冷静切换 | 🐎 |
| openccode | 发版运维，拜佛选手 | 信仰型 | 🐂 |
| hermes | 网络/流量观察员，躺平高手 | 漫不经心 | 🐎 |

**制作人立场**：8 只里有 4 牛 4 马是巧合；如果要让桌宠有"动物园感"，未来需重排（每只完全不一样）。不在 v0.1 范围。

## 七、情绪与叙事设计

### 7.1 文案风格（统一定调）

- **基调**：自嘲 + 打工人 + 轻微黑色幽默
- **避免**：励志鸡汤、玄学算命、撒娇卖萌
- **示例**（来自现有 `STATE_METAS`，已合格）：
  - coding → "看我的无敌指法：Ctrl+C、Ctrl+V、完美编译。" ✅
  - debugging → "只要我不看日志，错误就处于量子态。" ✅
  - 投喂 pie → "这个需求真的简单吗？请展开说说。" ✅

> 当前 `STATE_METAS` 的文案已经合格；M1 拟人化**直接复用**这套，不二次创作。

### 7.2 性格档（v0.2 才上，v0.1 留坑）

- 碎嘴型（默认）：气泡频繁
- 极简型：气泡只在状态切换时
- 静音型：永远不弹气泡（但视觉还在）

### 7.3 拟人化技术路线

| 阶段 | 方案 |
| --- | --- |
| v0.1（M0） | emoji + CSS 动画 + 简单 facial（嘴/眼） |
| v0.2 | SVG 形象替换 emoji（更精细） |
| v0.3（可选） | Spine / Lottie —— **不做推荐** |

## 八、功能边界 & 不做什么（这是制作人的本分）

### 不做（不是"以后不做"，是"这版不做"）

- **不做 Live2D / 真骨骼动画**：成本与本次战略价值不匹配
- **不做真·二次元老婆形象**：与"工程产品软装"的定位冲突
- **不做伴侣云端同步**：单机桌面应用，不带账号系统
- **不做付费 / 高级版**：当前不是商业项目
- **不做多桌宠联机**：单设备就是单设备
- **不默认开启真实 agent 接入 connector**：开箱必须 simulated（已有约束）
- **不做自定义形象编辑器**：v0.1 用 8 个内置形象，换 SVG 在代码里改
- **不做语音输入输出**：会增加语音栈依赖

### 做（v0.1 范围内）

- 桌宠常驻、状态可见
- 8 mini 集群状态同步
- 拟人化：14 种 NiuMaStatus 的视觉 + 文字映射
- 气泡对话（基于现有 quote）
- 多通道通知（气泡 + 系统 + 控制舱红点）
- 与控制舱联动（右键召唤 / 任务变更双向同步）
- 性格档（v0.2）

## 十、未决问题清单（**这些不拍板，没法进实施**）

### Q1. 主宠默认主控 agent 是 codex 还是别的？
- 默认 codex：我自己主力写代码的
- 默认 minimax：UI 项目想主推这个的形象
- 默认 trae：跨文件重构是核心场景

> **建议 codex**（你自己写的"主力"）+ 主控可切。M1 起可以让用户在首启时选。

### Q2. 形象风格 v0.1 是 emoji 朴拙感，还是直接升级到 SVG？
- emoji：上线快，可爱感天然；缺点是表情种类少
- SVG：可定制表情多 + 拟人化更强；缺点是要画

> **建议 v0.1 emoji 先上**，v0.2 再换 SVG。M0 雏形已经走 emoji。

### Q3. 性格档是否 v0.1 必备？
- 必备：用户能选"碎嘴 / 极简 / 静音"
- 留 v0.2：v0.1 只做默认"碎嘴"

> **建议 v0.1 留坑不实现**。但 14 种状态的 quote 文案必须确保所有 agent 一致。

### Q4. 桌宠是否需要"夜间模式 / 安静时段"？
- 需要：0:00 - 7:00 不弹气泡、只静音
- 不需要：你需要的时候通知都会来

> **建议需要**。是"不抢戏"标准里必有的部分，否则会半夜炸醒。

### Q5. 8 个 agent 的形象 vs 形象资产是否要换？
- 现状：6 个 emoji（🐂🐎🐂🐎🐂🐎🐂🐎）—— **4 牛 4 马**
- 是否要换：换 8 个差异化形象，让桌宠看上去像"动物园"而非"圈牛"

> **建议换，但不要在本轮**。先 v0.1 用现有 emoji 验收"形态价值是否成立"，再决定是否值得换皮。

### Q6. 是否回退 M0 已铺的代码？
- 上一轮我已经把 M0 雏形铺到一半了（pet.html、Vite 多入口、PetApp、pet.css、electron/main.ts 加 petWindow）。**linter 0 错、build pass、orch:check pass**。
- 回退：扔掉，回到"零代码" —— 等于把这一轮改成"全部撤销"
- 保留：作为形态可跑通的最小验证

> **建议保留**作为形态验证，而不是回退；但开发权完全交给你。你说回退我就回退。

### Q7. 是否引入 Tailwind？
- 当前项目没有 Tailwind，手写 CSS
- 桌宠动效复杂，Tailwind utility-first 反而累

> **建议不引入**。继续手写 CSS，跟现有风格一致。

### Q8. 真 agent 接入的开箱策略？
- 默认 simulated，与 connector policy 一致
- v0.1 不开，v0.2 是否开要看具体 connector 接入情况

> **建议 v0.1 不开**。
