# 多 Agent 牛马桌宠化开发计划 v0.1

> 标签：`[PM]#multi-agent-pet@v0.1`，与 `docs/orchestration/index.md` 角色体系一致。
> 本文是给产品/PM/开发的"开发计划 v0.1"，不是营销文案。
> 目标读者：项目 owner + 后续接手的实现者。

---

## 一、缘起：把"控制舱"升级为"桌宠"

### 现状盘点（已经有的，别从零造轮子）
- Electron 43 + Vite 6 + React 19 + TS + motion + lucide-react 桌面骨架
- 8 个 agent 角色 + 8 张头像（`icon/{codex,trae,qoder,minimax,workbuddy,openclaw,openccode,hermes}.{png,jpeg}`）
- 14 种 `NiuMaStatus`（idle/coding/debugging/meeting/coffee/testing/deploying/done/overtime/drinkingWater/panicking/slacking/praying/demanding），每种配 name + expression + 4 条 quote
- 5 种 `NiuMaAction`（投喂 pie / 打 coffee / 挥鞭 whip / 摸鱼 slack / 戳一下 poke）
- 主进程 IPC：getSnapshot / resetSeed / createTask / stopTask / clearCompletedTasks / applyNiuMaAction / onSnapshotChanged
- 本地 runner：spawn 子进程跑真实命令
- simulation runner：setInterval 推进 task 进度
- orchestration：PM + Supervisor + 短工的角色体系、`status.json`、`connectors.json`、`connectors.schema.json`
- 当前形态：1220×860 单窗口控制舱（启动可见）

### 桌宠目标（user 给的方向）
- 主动目的：**情趣价值 + 多 agent 监听 + 状态通知**
- 隐含需求：拟人化、有性格、能对话 / 弹气泡、能挂件 / 缩起常驻
- 隐含约束：要复用现有 snapshot / orchestration，不要从零搭

### 跃迁原则
- **不推翻**：现有 snapshot、IPC、runner、orchestration 都保留
- **不替换**：控制舱不删除；它和桌宠是"双形态共存"
- **不停 mock**：real sub-agent 仍以 connectors 走，不开箱默认启用
- **不堆技术债**：lint 0 错误基线不能被搞坏

## 二、产品形态定义（要按这个造，不是"差不多就行"）

### 1. 双形态：主桌宠（默认常驻） + 控制舱（托盘召唤）
| 项 | 主桌宠 | 控制舱 |
| --- | --- | --- |
| 入口 | 开机自动弹 | 托盘菜单 / 主宠右键 → "调出控制舱" |
| 形态 | 一只漂浮在桌面的可拖拽小生灵 | 1220×860 完整控制台 |
| 持续性 | 长期可见，不抢焦点 | 按需调出，可隐藏到托盘 |
| 任务 | 日常陪伴 + 状态通知 + 主动状态总览 | 详细管理：建任务、查日志、看任务队列 |

### 2. 主桌宠规格
- 形象：当前"主控 agent" 的 emoji 头像（默认 `codex`，可右键切换）
- 尺寸：默认 180×180，可缩 64~240
- 位置：可拖、可贴边（吸附 top/left/right/bottom）、可置顶 alwaysOnTop
- 行为：idle 周期动效（呼吸 / 眨眼 / 偶尔躺平），状态变更时触发反应动效（欢呼/叹气/打滚）

### 3. 副宠集群（8 mini pet）
- 默认显示：8 个 mini 头像（40×40）排成一行在主宠右上角
- 订阅：每个 mini pet 对应一个 agent；状态 idle/active/thinking/error 反映实时
- 行为：
  - active 小宠头顶冒小风扇
  - error 小宠冒烟 + 抖一抖
  - done 小宠蹦一下 + 撒花粒子
  - 长时间 idle 小宠打盹（zzz）

### 4. 状态气泡
- 主宠头顶 / 旁边对话泡，自动出现 N 秒后收回
- 内容来源：
  - 当前 status 的 quote 文案（已存在）
  - 任务完成 / 失败的实时通知（"codex 完成了『清理日志』"）
  - 主动戳一下按钮的搞怪回应
- 双层结构：bubble + speech trail（包边带尾巴）

### 5. 桌面通知（系统级）
- 任务结束 / error / PM 通知 → Electron `Notification`
- 通知偏好可在控制舱开 / 关（按类别关）

## 三、核心设计决策（不是清单，是已经定了）

| # | 决策 | 取舍理由 |
| --- | --- | --- |
| D1 | 单主宠 + 8 副宠集群，**不是每只都大** | 单只拟人感最强 + "宝宝圈"最直观呈现 8 个 agent；每只都是大桌宠会变成桌面灾难 |
| D2 | 主窗用 `frame:false, transparent:true, alwaysOnTop, skipTaskbar, resizable:false → resize 通过 renderer`, `setIgnoreMouseEvents({ forward: true })` 让透明区穿透 | 这是 Electron 桌宠的"标准装弹方式"，最稳 |
| D3 | 拟物化技术栈 = motion + lucide-react + CSS keyframes + 像素态 SVG，**不上 Live2D** | Live2D 体积大、动画成本高，不适合第一阶段；CSS+SVG 已足够"有灵魂" |
| D4 | 状态源三层订阅：① `AgentSnapshot`（task/runtime）② `OrchestrationStatus`（PM/Supervisor 短工 lanes）③ 用户事件（拖拽/右键） | 三层是真正"听到所有 agent 状态"的关键，缺一层就退化成普通宠物 |
| D5 | 状态通知 4 通道：内嵌气泡（主）+ Electron Notification（系统）+ 控制舱右上红点 + 可选音效 | 只气泡会丢；只系统通知会被关机；多通道叠加才"靠谱地被感知到" |
| D6 | 与控制舱 = 双形态共存，**不是替代**。桌宠常驻 = 默认行为；控制舱 = 托盘入口 | 控制舱是天生的"精管"，强行合一会阉割管理能力 |
| D7 | demo 走 simulated runner（已存在），**真实 agent 仍按 connector policy 默认 disabled** | 与 `connectors.json` 不冲突；保证无真实 CLI 也能完整情绪循环 |
| D8 | 复用 `NiuMaAction` 作为右键菜单 → 投喂/打咖啡/挥鞭/摸鱼/戳 | 不发明新交互，存量文案 + 表情直接盘活，桌宠立刻"可玩" |
| D9 | 所有桌宠视觉资源放 `src/pet/`，不放 `src/components/` | 跟现有的 NiuMaWorkspace 物理隔离，避免控制舱渲染时被拖慢 |
| D10 | 跨窗口通信全部走 IPC，**不直接共享 React Context** | 桌宠窗口和控制舱窗口可能在不同时刻出生，需要一个明确的事件总线 |

## 五、技术架构（M0 起就要想清楚）

### 进程分层

```
┌─────────────────────────────────────────────────────────────┐
│ Electron Main (`electron/main.ts` 扩展)                        │
│  - createPetWindow()                                          │
│  - 拖拽边界（snap to edge）                                   │
│  - setIgnoreMouseEvents 透传控制                              │
│  - IPC handler: pet:open/hide/set-active-agent/set-scale      │
└─────────────────────────────────────────────────────────────┘
                          ▲          ▼
                   contextBridge (`electron/preload.ts`)
                          ▲          ▼
┌─────────────────────────────────────────────────────────────┐
│ Pet Renderer (`src/pet/`)                                      │
│  PetApp.tsx                                                   │
│   ├─ PetBody.tsx        ← 主宠本体                            │
│   ├─ PetBubble.tsx      ← 头顶气泡                              │
│   ├─ MiniPetCluster.tsx ← 8 个 mini pet                       │
│   │   └─ MiniPetBadge.tsx                                    │
│   ├─ PetWakeReaction.tsx ← 通知反馈动效                         │
│   └─ hooks/                                                  │
│      ├─ useDragAndSnap.ts  ← 拖拽 + 贴边                       │
│      ├─ usePetNotification.ts ← 气泡/系统通知                  │
│      └─ useActiveAgent.ts  ← 当前主控 agent                    │
└─────────────────────────────────────────────────────────────┘
         ▲       ▲
         │       │  共享 AgentSnapshot（IPC + main.ts 已实现）
         │       └─ 共享 OrchestrationStatus（main.ts 新增读）
         └─ 控制舱（src/App.tsx）可继续独立订阅同一 snapshot
```

### 新增 IPC（不进 stage 的话不用全做，M0 起步用最小集）

| Channel | 用途 | 阶段 |
| --- | --- | --- |
| `pet:open` | 弹起主宠窗口 | M0 |

| `pet:toggle-control-cockpit` | 从桌宠右键开关控制舱 | M1 |
| `pet:set-active-agent` | 切换主宠显示的 agent | M1 |
| `pet:tick-notification` | 任务状态变更触发的气泡事件 | M3 |
| `pet:set-snapped-edge` | 拖拽贴边落位 | M0 |

### PetApp 子树（最终交付）

## 六、6 阶段开发路线

### M0 — Pet Window 雏形（**今天就交付**）
- `electron/main.ts` 新增 `createPetWindow()` + `petWindow?: BrowserWindow`
- `petWindow` 470×360 默认；`frame:false, transparent:true, alwaysOnTop:true, skipTaskbar:true, resizable:false`
- preload 加 `pet:open`, `pet:hide`
- 新增 `src/pet/PetApp.tsx` 渲染占位但**能看见**的"生物"（一个能动的 emoji 圆脸）
- 启动 `npm run dev` 时主宠窗口 + 控制舱窗口**都**起来；控制舱默认藏托盘

#### M0 验收
- [ ] `npm run lint` 通过
- [ ] `npm run build` 通过
- [ ] `npm run orchestration:check` 通过
- [ ] 启动桌面后看到一个会动的圆脸漂浮在屏幕一角
- [ ] 关闭时能进托盘；右键托盘能"显示主宠"
- [ ] 不影响现有 NiuMaWorkspace / IPC

### M1 — 状态 ↔ 拟人化映射（1~2 天）
- 落 `petPersonality.ts`，13 种 status → 视觉 + 表情
- 主宠接入 AgentSnapshot：`runtime[activeAgentId].status`
- idle/focus/notify 三档动画（呼吸、眨眼、抖一抖）
- 右键菜单：切换 active agent / 投喂/咖啡/摸鱼/戳

#### M1 验收
- [ ] 主宠当前状态变化时肢体 / 表情即时切换
- [ ] 模拟任务跑起来，主宠能看到 coding → done 全流程
- [ ] 右键切 active agent 无需重启窗口

### M2 — 副宠集群
- `MiniPetCluster`：8 mini pet 排成 1×8
- 订阅 `agents[].status` + 当前每 agent 的 running task 数
- mini 宠的 4 状态：idle / active / error / done
- 状态切换时跑反馈（欢呼/叹气/打滚）+ 鼠标 hover 显示 agent 名

#### M2 验收
- [ ] 8 mini 宠全部渲染；点击任一 = 切换主宠为该 agent
- [ ] 模拟任务下发时，codex mini 立即 active；任务完成时蹦 + 出气泡

### M3 — 状态通知通道
- `usePetNotification` 监听 snapshot 增量
- 三通道：
  - 内嵌气泡（主宠头顶）
  - Electron `Notification`（系统）
  - 控制舱未读数红点
- 偏好可在控制舱开 / 关

#### M3 验收
- [ ] 真下发任务后三个通道按预期出现
- [ ] 关闭通知偏好 → 通知不弹

### M4 — Orchestration 接入
- main 进程读 `docs/orchestration/status.json` 喂给 renderer
- 主宠在出现 PM 卡 / Supervisor blocker 时切换到一个特殊状态"指挥部"
- 副宠集群旁边加一个"短工状态带"：列 PM / Supervisor / 各 lane 的状态

#### M4 验收
- [ ] 编辑 `status.json` 并触发刷新 → 主宠 / 集群视觉同步变化
- [ ] 出现新 lane 时集群缩放不挤

### M5 — 拟物化升级（用户偏好形）
- 多套主题：默认 / 极简 / 高饱；可切
- 性格档：碎嘴型 / 极简型 / 静音型（控制气泡文案密度）
- 鼠标 hover 时主宠有"被注意"反应

#### M5 验收
- [ ] 切主题不需要重启窗口
- [ ] 性格档把气泡文案密度按档位调整

## 八、风险与回退

| 风险 | 影响 | 回退 |
| --- | --- | --- |
| Electron 拖拽 + 透明穿透在 Win11 缩放场景异常 | 主宠定位抖动 | 退回简单 `BrowserWindow` + 自绘点击区 |
| motion 频繁 re-render 让桌宠动画卡顿 | 主宠掉帧 | 降级到 CSS keyframe 替代 motion 部分 |
| Orchestration `status.json` 文件读到 renderer 触发 IPC 风暴 | 卡顿 | 改为 startup 一次 + 1s 节流事件 |
| 真实 agent 接入时主宠通知刷屏 | 被打扰 | 加每分钟 N 条上限 + 折叠 |
